Reviewed Rules

Rule definitions, severity levels, and evaluation purpose used by the analyzer.

Hardcode

Severity 3

Detects credentials, tokens, secrets, API keys, bearer tokens, signatures, email addresses, or other sensitive information defined as fixed values inside the flow, variables, or action properties, instead of being resolved through a secure mechanism or external configuration. This practice increases the risk of information exposure and makes the solution harder to manage securely.

Parametrizable

Severity 2

Detects non-sensitive fixed values defined directly in the flow, such as paths, endpoints, file names, configuration identifiers, or operational values, which should be managed through parameters, environment variables, connection references, or external configuration. Its purpose is to improve maintainability, reusability, and promotion across environments.

Manejo de errores (RunAfter)

Severity 2

Detects relevant flow actions that do not include an explicit error-handling configuration through RunAfter, or an equivalent exception-handling structure. In Power Automate Cloud, RunAfter is recommended to define behavior for states such as failure, cancellation, or timeout, in order to keep execution controlled and traceable.

Retrasos

Severity 2

Detects delay, wait, or interval actions configured with literal or fixed values inside the flow. The use of delays should be kept to a minimum because it can affect performance and make the automation fragile; when delays are necessary, their use should be documented and their value should be obtained from a parameter.

Nomenclatura de actividades

Severity 1

Detects actions with default names, non-descriptive names, or names that do not comply with the writing rules defined for Power Automate Cloud, such as the use of accents, the letter ñ, special characters, or unnecessary numbering. The action name should be clear, descriptive, and consistent with the functionality it performs.

Nomenclatura de variables

Severity 1

Validates that variable names follow the Cloud naming standard by using UpperCamelCase and a prefix that represents the data type, such as Bln, Int, Flt, Str, Obj, or Arr. Names should be unique, readable, and consistent with their purpose within the flow.

Prefijos variables / parámetros

Severity 1

Validates that parameters exchanged between flows use the correct prefix according to their direction: in_ for input, out_ for output, and io_ for input/output, followed by the data type and a descriptive name. In addition, the first letter after the prefix must follow the convention defined in the manual to maintain consistency and understanding across related flows.

Nomenclatura de flujos

Severity 1

Validates that flow names follow a clear, uniform, and consistent convention aligned with the action they perform, using UpperCamelCase, without spaces, accents, or special characters. The name should make the flow’s purpose easy to identify and preserve consistency within the solution.

Condición IF

Severity 1

Detects If conditional structures whose main branch is empty, lacks useful actions, or is organized in a way that affects flow readability and understanding. This validation aims to highlight poorly structured conditions that make maintenance harder or create unclear logic inside the automation. The manual also limits condition nesting to preserve readability, although in Cloud this validation may require justification depending on the case.

Comentarios

Severity 1

Validates whether flow actions and scopes include enough notes or descriptions to make them easier to understand and maintain. In Power Automate Cloud, it is recommended to add descriptions to actions and scopes, aiming for at least 33% of actions to include visible documentation inside the flow.
